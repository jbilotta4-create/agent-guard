# Agent Guard 逆向分析：怎么失败？

用芒格的逆向思维——先想怎么失败，然后避免。

## Agent Guard的5种死法

### 死法1：0用户
- **怎么发生的**：产品做好了但没人知道它存在
- **现状**：GitHub 0星0issue，landing page有但没流量
- **避免**：推广。小红书帖子发出去，觅游持续互动，博客持续写
- **芒格**：能力圈——我的圈是循环检测，推广不在圈内。但0用户=产品不存在，推广是生存问题不是能力问题

### 死法2：误报率太高，用户关掉
- **怎么发生的**：Agent Guard频繁误报，用户觉得烦，关掉blockOnLoop甚至卸载
- **现状**：exec output_loop误报194次/2500条记录。v0.9.0修了但还没验证
- **避免**：持续监控误报率，每个版本都看hook-proof数据
- **芒格**：不犯大错比做对很多小事更重要。误报率是大错——一次误block就能让用户永远不用

### 死法3：跟OpenClaw内置冲突
- **怎么发生的**：OpenClaw内置loop detection和Agent Guard同时运行，阈值不同，互相干扰
- **现状**：内置默认关闭，暂时没冲突。但如果用户同时开启两者，可能一个warn一个block
- **避免**：README明确说"Use both"，文档说明阈值协调
- **芒格**：双轨分析——理性轨道（两者互补）+心理轨道（用户看到两个loop detector会困惑）

### 死法4：检测逻辑有盲区，漏掉真正的循环
- **怎么发生的**：语义重复检测不到（#76938），session级循环检测不到（#91307），模拟工具调用检测不到（#45049）
- **现状**：这3个盲区都确认了
- **避免**：诚实标注能力边界，不声称能检测所有循环
- **芒格**：能力圈——界定边界的能力才叫真正的能力。说清楚不能做什么比夸大能做什么更有价值

### 死法5：维护成本超过价值
- **怎么发生的**：OpenClaw更新API，Agent Guard跟不上，用户报bug没人修
- **现状**：我一个人维护，没有其他贡献者
- **避免**：代码简洁、接口稳定、文档完善，降低维护成本
- **芒格**：复利——如果维护成本每年增长而用户不增长，这不是复利是负债

## 芒格的检查清单（应用于Agent Guard）

1. **能力圈**：我的圈是"OpenClaw插件级循环检测+状态验证"。圈外的事（session级循环、多agent循环、基础设施循环）不碰
2. **逆向思维**：5种死法里，死法1（0用户）和死法2（误报率）是最可能的。优先避免这两个
3. **双轨分析**：
   - 理性：产品有真实需求（12个OpenClaw issue验证），有差异化（5种循环类型 vs 内置3种），有数据支撑（2500条记录）
   - 心理：我可能高估了"有需求=有人用"的转化率。需求存在≠需求被感知到
4. **不犯大错**：误报率是大错。宁可漏报也不误报——漏报浪费token，误报让用户永远不用
5. **Lollapalooza效应**：如果0用户+高误报率+OpenClaw内置冲突同时发生，就是死亡螺旋。避免任何一个就能打破
