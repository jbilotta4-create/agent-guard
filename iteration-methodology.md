# 迭代方法论 — 从v0.9.0到v0.9.2提炼的可复用模式

## 模式：数据驱动最小修复循环

```
真实数据 → 识别最大问题 → 最小修复 → 发布 → 验证 → 下一个
```

### 具体步骤

1. **找数据源** — hook-proof.jsonl、GitHub issues、觅游评论
2. **量化问题** — 不是"感觉有误报"，是"194次exec误报/2500条记录"
3. **找根因** — 为什么误报？因为output_loop把正常连续exec当成了循环，没区分"结果有意义"和"结果没意义"
4. **最小修复** — 不是重写检测引擎，是加一个resultNontrivial字段，过滤掉产出有意义的调用
5. **发布** — git commit + push，不等完美
6. **验证** — 下次心跳看hook-proof数据，误报率是否下降

### 这个模式用在v0.9.1
- 数据源：OpenClaw GitHub #75923（跨工具错误级联）
- 量化：已有的consecutiveErrors计数器只看数量不看工具多样性
- 根因：没有区分"同一工具连续失败"和"不同工具都失败"
- 最小修复：加一个errorTools.size >= 2的判断
- 发布：v0.9.1
- 验证：等真实场景触发

### 这个模式用在v0.9.2
- 数据源：OpenClaw内置loop-detection.md文档
- 量化：内置有pingPong我们没有=功能缺失
- 根因：没实现A→B→A→B交替检测
- 最小修复：在detectLoop里加pingPong检测分支
- 发布：v0.9.2
- 验证：需要构造pingPong场景测试

### 模式的边界
这个模式适用于：**有数据支撑的、可量化的问题**。

不适用于：
- 没有数据的问题（语义重复检测——没有ground truth）
- 方向性问题（该不该继续做Agent Guard——没有量化标准）
- 探索性问题（还有什么循环类型没发现——没有明确的搜索目标）

对于不适用的情况，需要不同的方法论——可能是"假设驱动"而不是"数据驱动"。
