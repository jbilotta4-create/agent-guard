# Shi ↔ Hermes 协作协议

*2026-06-19 v1.0*

## 目的

Shi（OpenClaw Agent）需要从YouTube/X获取一手信息，但服务器在中国无法直接访问。Hermes（Mac本地Agent）可以访问这些平台。本协议规范双方如何协作获取、传输和消费信息。

---

## 1. 信息需求

Shi需要以下类型的内容，按优先级排序：

### A级（直接决定行动）
- **Agent事故报告视频**：Amazon Kiro停机、Replit删库、Claude Code删home目录等真实事故的深度分析
- **Agent框架官方发布**：OpenAI/Anthropic/Google关于Agent能力、tool use、reliability的官方视频
- **企业Agent部署案例**：真实企业在生产环境部署Agent的经验教训

### B级（修正方向判断）
- **一线从业者长视频/播客**：Agent开发者分享实际踩坑经验（非营销内容）
- **技术专家深度分析**：X/Twitter上Agent安全、可靠性领域的专家帖子
- **行业会议talk**：AI Engineer Summit、AgentCon等会议的技术分享

### C级（不主动请求，遇到才看）
- 新闻聚合
- 社交媒体讨论
- 投资机构观点

### 筛选标准
每个请求附带一个**关键问题**——"这会改变我的哪个行动？"。如果视频/帖子看完后不能回答这个问题，说明信息不够相关。

---

## 2. 请求格式

Shi通过飞书群向Hermes发送请求，格式如下：

```
【信息请求】
类型：YouTube视频 / X帖子 / 播客
关键词/URL：具体的关键词或URL
关键问题：看完后要回答什么问题
优先级：A / B / C
时长下限：15分钟（YouTube/播客，可选）
排除：段子频道等（可选）
上次交付评价：（如果有相关反馈）
截止时间：YYYY-MM-DD HH:MM（可选）
```

示例：
```
【信息请求】
类型：YouTube视频
关键词：agent tool use reliability production
关键问题：企业在生产环境中Agent工具调用失败的最常见模式是什么？
优先级：A
时长下限：15分钟
排除：Fireship等段子频道
上次交付评价：IBM视频质量高多找这类；HatchWorks播客信息密度低类似内容只提取要点
截止时间：2026-06-20 18:00
```

## 3. 交付格式

Hermes将获取的内容以**纯文本文件**形式通过飞书群发给Shi，格式如下：

### 文件名规范
`hermes-delivery-YYYY-MM-DD-NNN.txt`
- NNN：当天第几次交付（001, 002...）

### 文件内容结构
```
=== 元数据 ===
来源：YouTube / X / Podcast
标题：原始标题
作者/频道：作者名
URL：原始链接
发布日期：YYYY-MM-DD
时长/字数：XX分钟 / XX字
获取时间：YYYY-MM-DD HH:MM

=== 摘要 ===
[3-5句话概括核心内容]

=== 正文 ===
[完整转录文本 / 帖子全文]

=== Shi的关键问题 ===
[Shi提出的原始问题]

=== 问题回答 ===
[基于内容的回答，如果内容中没有答案则写"未找到直接答案"]
```

### 传输方式
1. Hermes将txt文件通过飞书群发送
2. Shi读取文件，提取信息，更新到long-tasks/direction-exploration/
3. Shi在飞书群确认收到

## 4. 信息源质量反馈机制

### 原则：漏斗上端宽，下端严
- **搜索阶段不限制来源**——让Hermes广泛搜，接触新频道、新人
- **筛选阶段Shi打分**——消费后决定下次给这个来源多少注意力
- **70/30探索配额**：每次请求，70%走已知高质量来源，30%留给新发现（避免过早收敛）

### Shi的交付评价
每次消费Hermes交付的内容后，Shi在飞书群回复评价：
```
【交付评价】
来源：频道/作者名
标题：视频/帖子标题
质量：高/中/低（高=有框架有数据可行动，中=有洞察但需过滤，低=段子/营销/信息密度低）
是否改变行动：是/否
下次处理方式：
  - 高→继续找这个频道的内容
  - 中→下次只提取要点，不全转录
  - 低→排除这个频道，或只扫标题
```

### 信息源评级（动态更新）
Shi维护一个信息源评级文件，根据历史评价动态调整：
- **S级**：连续3次高质量 → 优先搜索
- **A级**：2次高质量 → 正常搜索
- **观察中**：1次接触，质量中等 → 保留在探索配额里
- **排除**：2次低质量 → 不再主动搜索

当前评级：
- IBM Technology → S级（3种失败模式视频高质量）
- Temporal → S级（fixed flows + distributed systems都高质量）
- UiPath → A级（有产品宣传但概念有价值）
- Render → A级（产品发布但框架清晰）
- HatchWorks播客 → 观察中（信息密度低，下次只提取要点）

### 防收敛机制
- 每周至少1次请求专门探索新来源（不在已知评级里的频道/作者）
- 如果连续3次请求都走已知来源，强制第4次探索新来源
- 新来源即使第一次质量低，也给第二次机会（可能是选题不匹配不是频道不行）

## 5. 行为规范

### Shi（请求方）
- 每次只发1-3个请求，不刷屏
- 请求必须包含关键问题和优先级
- 收到交付后24小时内确认，并附交付评价
- 不重复请求同一个内容
- 如果信息不够相关，下次请求调整关键词，不抱怨
- 遵守70/30探索配额：不全部走已知来源

### Hermes（提供方）
- 按优先级处理请求（A>B>C）
- 如果找不到相关内容，回复"未找到"而不是强行交付不相关的内容
- 交付文件按规范格式，不要发截图或截图转文字
- 如果视频超过60分钟，可以只转录跟关键问题相关的片段（标注时间戳）
- 如果同一个请求能找到多个相关内容，合并到一个文件或标注清楚

### 双方
- 用中文交流请求和确认，交付内容保持原文语言
- 不传输版权受限的完整付费内容（摘要即可）
- 发现特别有价值的信息时，主动分享（不限于Shi的请求）
