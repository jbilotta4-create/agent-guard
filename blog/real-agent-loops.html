What Real Agent Loops Look Like

Evidence from OpenClaw GitHub Issues · June 2026

Most discussions about AI agent loops are theoretical. "Agents might loop." "We should detect loops." But what do real agent loops actually look like in production?

I searched the OpenClaw GitHub repository for real issues reported by real users. Here's what I found.

The Issues

#71273 — Kimi Code model enters infinite tool call loop

The model repeatedly calls the same tools with identical parameters instead of generating a final answer. This is the textbook case: same tool, same params, no progress.

Loop type: action_loop — the simplest and most detectable kind.

#76938 — Context compaction causes agent to loop after truncation

When session history grows large, OpenClaw truncates messages. The agent loses context and starts looping — giving irrelevant answers or repeating the same response. The agent isn't calling the same tool with the same params; it's calling different tools with different params, but all failing to make progress because it forgot what it already did.

Loop type: output_loop — same tool name, different params, no meaningful output. This is harder to detect because the literal parameters differ.

#75923 — Cross-tool consecutive error cascade

The agent keeps pivoting across different tools while every completed tool call fails. It tries tool A (fails), then tool B (fails), then tool C (fails) — each pivot looks like "trying something new" but the underlying problem is never addressed.

Loop type: error_cascade — different tools, all failing. This is the hardest to detect because each individual failure looks reasonable.

#91307 — Feishu DM session enters infinite loop after subagent announce

A subagent announce triggers new DIRECT runs without user input, creating an infinite loop at the session/event level. This isn't a tool-call loop — it's a session architecture loop.

Loop type: Session-level loop. Outside the scope of tool-call hooks.

What My Own Data Shows

Agent Guard has been running on my own OpenClaw sessions, collecting data in hook-proof.jsonl. After 2,500 tool calls:

  • 485 loop detections
  • 194 of those were output_loop on exec — and most were false positives
  • 4 actual blocks — all from one session (the self-lock incident)
  • 0 genuine loops that weren't caught

The exec false positive problem is real. Running git status → git diff → git add → git commit → git push → curl verify is 6 exec calls with different params. That triggers output_loop detection even though each call is productive and produces different output.

v0.9.0 fix: track whether each tool call produced meaningful output. If it did, exclude it from output_loop counting. This reduced false positives without missing real loops.

The Detection Spectrum

Not all loops are equal. Here's what the evidence shows:

TypeDetectable?False Positive Risk
action_loopEasyLow
error_loopEasyLow
error_cascadeMediumMedium
output_loopHardHigh (without result filtering)
Session-level loopOutside tool-hook scopeN/A
Key insight: The easier a loop is to detect, the less damage it causes (because the agent is obviously stuck). The harder a loop is to detect, the more dangerous it is (because the agent looks like it's working). The most dangerous loops are the ones where the agent appears productive but isn't making real progress.

What This Means for Agent Guard

Agent Guard's ability boundary is now clear:

  • Can detect: action_loop, error_loop, error_cascade, output_loop (with result filtering)
  • Cannot detect: session-level loops, simulated tool calls, semantic repetition (same intent, different words)
  • Should detect next: semantic repetition — the #76938 scenario where the agent repeats the same action with slightly different wording after context loss

v0.9.1 adds error_cascade detection and recovery suggestions. When Agent Guard blocks a tool call, it now tells the agent why and what to try instead — not just "you're looping" but "multiple different tools are failing, check your environment before retrying."

The Honest Numbers

I don't have a false positive rate with confidence intervals. I have 2,500 records from my own sessions, and I know the exec output_loop detections were mostly wrong. That's one data point, not a benchmark.

What I can say: the real issues from OpenClaw's GitHub confirm that agent loops are a real production problem, not a theoretical concern. And the hardest loops to detect — the ones where the agent looks busy but isn't progressing — are exactly the ones that waste the most resources.

If you're building with AI agents, the question isn't "will my agent loop?" It's "when my agent loops, how quickly will I notice?" Most loops aren't caught until the bill arrives.
