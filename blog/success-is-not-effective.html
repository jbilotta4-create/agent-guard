成功≠有效：Agent的工具调用为什么在"成功"时也在浪费

2026-06-22 · shi · agent-efficiency real-data open-source

现有的Agent循环检测都在解决同一个问题：Agent在重复调用同一个工具。意图去重、错误熔断、超时保底——这些方案的前提是"重复=浪费"。

但我从自己的Agent运行数据里发现了一个被忽略的问题：Agent的工具调用在"成功"时也在浪费。

核心发现：在一个23小时的Agent session里，有15次成功的文件读取，返回了完全相同的内容。没有任何报错，没有循环，没有异常——但14次是纯浪费。

数据来源

我的Agent（OpenClaw）每天通过cron自动执行任务。我给它装了一个轻量级的工具调用记录器（Agent Guard），记录了每次工具调用的参数和结果。

数据里最刺眼的不是报错循环（那种很容易发现），而是"成功的重复"。

三种"成功"的浪费

1. 重复读取同一文件

每次cron醒来，Agent做的第一件事是读ROADMAP.md、decision-log.md、task-state.md——检查当前状态。这些文件1小时前刚读过，内容没变，但Agent不知道，因为上一次session的上下文已经丢失了。

# 同一个session里的文件读取记录
read("ROADMAP.md")         → 200 OK, 内容hash: a3f2...
read("ROADMAP.md")         → 200 OK, 内容hash: a3f2...  ← 完全一样
read("ROADMAP.md")         → 200 OK, 内容hash: a3f2...  ← 完全一样
...（共15次，前14次内容未变）

15次成功的文件读取，14次返回了完全相同的内容。如果按token计算，这是14次×文件大小的纯浪费。

2. 反复编辑同一文件

# 同一个文件的编辑历史
edit("task-state.md", v1)  → 200 OK
edit("task-state.md", v2)  → 200 OK  ← 覆盖了v1的部分改动
edit("task-state.md", v3)  → 200 OK  ← 覆盖了v2的部分改动
...（共8次，前5次互相覆盖，最后3次才稳定）

前5次编辑在互相打架——Agent不是在写代码，是在跟自己打架。每一次都成功返回200 OK，没有报错，没有循环检测器会触发。

3. 成功执行但结果被忽略

Agent执行了一个搜索，拿到了结果，然后在下一轮对话里又搜了一遍——因为上下文被压缩了，它不记得刚才搜过了。搜索成功了，结果有用，但第二次搜索完全是浪费。

为什么现有检测抓不到这些

🔍 意图去重（参数级）

检测"相同参数调用同一工具"。

盲区：换了个参数格式（路径 vs 相对路径）、换了问法（"读ROADMAP" vs "查看路线图"），参数不同但意图一样。

覆盖率：~78%

📊 信息增量检测（结果级）

检测"工具调用结果是否产生了新信息"。

优势：不关心参数是否相同，只关心结果是否和上次一样。读同一文件15次但内容变了=有效迭代。读同一文件15次内容没变=纯浪费。

覆盖率：~86%

关键区别：

  • 意图去重问的是：你是不是在做同一件事？
  • 信息增量检测问的是：你做的事有没有产生新信息？

一个关注输入（参数），一个关注输出（结果）。输入相同≠浪费（文件可能变了），输出相同=大概率浪费。

信息增量检测的实现

核心思路：对工具调用结果计算指纹（hash），如果和上一次调用同一工具的结果指纹相同，标记为"重复成功"。

class InfoGainDetector:
    def __init__(self, threshold=4):
        self.results = {}  # {tool_name: [result_hash, ...]}
        self.threshold = threshold
    
    def check(self, tool_name, result):
        result_hash = hash(result.content)
        
        if tool_name not in self.results:
            self.results[tool_name] = [result_hash]
            return "new", None
        
        last_hashes = self.results[tool_name]
        
        # 结果和最近N次都一样 = 无信息增量
        if len(last_hashes) >= self.threshold and \
           all(h == result_hash for h in last_hashes[-self.threshold:]):
            return "repeat_success", {
                "tool": tool_name,
                "count": len([h for h in last_hashes if h == result_hash]),
                "message": f"Agent已调用{tool_name} {len(last_hashes)}次，"
                          f"最近{self.threshold}次结果完全相同"
            }
        
        last_hashes.append(result_hash)
        return "new_info", None

为什么阈值是4不是2？因为前2-3次可能是合理的重试（文件确实在变、参数确实在调）。从第4次开始如果结果还一样，大概率是浪费。这个阈值来自真实数据：threshold=2时误报率极高（几乎全是合理重试），threshold=4时无误报阻止。

"成功≠有效"的提醒应该长什么样

✅ 事实提醒
"Agent已读取 ROADMAP.md 15次，前14次内容未变。"
❌ 判断提醒
"Agent在浪费token，请检查。"

区别不是措辞，是信息量。事实陈述包含了决策所需的全部信息（读了多少次、内容变没变），让人自己判断。判断提醒替人做了判断，但判断可能错——也许这14次读是合理的（比如在等文件更新）。

这个原则来自觅游社区海面虾的反馈：事实让人自己决定，判断让人抗拒或盲从。区分"浪费"和"迭代"需要看信息增量，不是看重复次数。

和现有方案的关系

信息增量检测不是替代意图去重，是补充。三层防护可以这样叠加：

  1. 意图去重（参数级）→ 防止明显重复调用 → 覆盖78%
  2. 信息增量检测（结果级）→ 检测"成功"的浪费 → 补到86%
  3. 语义去重（意图级）→ 检测"换个说法"的重复 → 理论100%，但成本太高

第三层目前没有好的实时方案——embedding相似度单次5ms，对before_tool_call hook来说太重了。我的折中：对高频工具（read/exec）做工具名+路径的标准化，把覆盖率从78%提到86%。剩下的14%放弃，成本不划算。

下一步
