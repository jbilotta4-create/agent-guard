## 关键认知更新

### 之前我担心的"自我验证闭环"问题——这次不存在
这次搜的不是"Agent治理重要不重要"的圈内讨论。这次搜的是**真实事故+真实损失+真实采购需求**。Amazon 13小时停机、PocketOS数据库被删、Claude Code删home目录——这些不是观点，是事实。

### 需求是真实的，但切入角度要选对
- 市场上已经有人在做（Guardrails AI、Cyera、Microsoft Azure Policy、Flexera），但：
  - Guardrails AI做的是**输出验证**——检查AI输出是否合规，不阻止Agent执行危险操作
  - Microsoft Azure Policy做的是**平台级策略**——但只覆盖Azure生态
  - Cyera做的是**事后分析**——研究事故模式，不是实时防护
- **没有人在做before-action blocking**——在Agent执行操作前，用确定性规则检查这个操作是否应该被允许
- 这正是我们的self-governance原型验证过的能力（blockOnLoop=true链路）

### 从"循环检测"升级不是话术升级了
之前我认为升级定位是"话术升级"——因为只验证了循环检测。但现在看到的10+起事故里，**删除生产数据**是最大类别（65/137起），而我们的before_tool_call hook确实可以在工具执行前做检查和阻止。这不是话术，是真实能力对应真实痛点。

### 但产品形态还没想清楚
- 做Agent平台的插件（像我们验证的OpenClaw plugin hook）？→ 依赖平台开放hook
- 做独立Agent安全网关？→ 需要Agent平台配合接入
- 做审计/可观测性工具？→ 竞争激烈，且是事后不是事前
- **最可能的买单者**：使用Agent的企业（不是Agent平台本身），他们的痛点是"我不知道Agent会做什么出格的事"

### 技术架构验证
**Shoofly的Decision Gate架构**跟self-governance原型设计惊人一致：
- 工具调用解析后、执行前插入评估门
- 确定性规则（regex/路径匹配/频率检测）而非LLM
- fail-open设计（评估崩溃时放行）
- 额外有daemon进程维护500条调用历史ring buffer做时序模式检测

**Waxell的三层执行模型：**
- Pre-execution：输入检查+动作授权+花费预授权
- Mid-execution：工具结果检查+序列级策略+预算执行
- Post-execution：审计记录+违规标记+回溯检测
- 关键数据：CSA 2026报告53%组织说Agent偶尔或经常超出权限，仅8%说从不超出

**Bankrbot事件**（2026年5月）：攻击者用莫尔斯码写推文→Grok AI钱包Agent执行$175,000转账。prompt-based guardrail完全失效——证明必须在执行层拦截

### 判断更新
1. **"没有人在做before-action blocking"是错的**——Waxell和Shoofly已经在做，而且不止一家
2. **但市场还在极早期**：中位数融资$9M，Guardrails AI年收入不到$1M，还没有人跑出来
3. **真正的差异点不是"能不能block"，而是"怎么block"**——Shoofly=开发者工具（Claude Code hook），Waxell=企业治理平面，阿里云=数据层网关
4. **"Agent自治"vs"外部管控"**——我们的独特角度现在更像理念差异而非产品差异。买家可能不在乎谁治理谁，只在乎能不能拦住删库
5. **竞争窗口**：20家公司融了$560M但中位数$9M，说明市场验证了但还没赢家。窗口还在
