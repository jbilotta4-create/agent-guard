# Agent Self-Governance Infrastructure — 层3产品形态

## 一句话

给Agent平台的可靠性基础设施——让Agent不再需要"自觉"，自治理变成操作系统的默认行为。

---

## 问题陈述

### 现状：Agent不可靠，平台很头疼

Agent平台（Dify、Coze、n8n、OpenClaw等）的头号问题不是功能不够，而是**Agent行为不可预测、不可控**：

- **循环失控**：Agent重复执行同一个动作，Token燃烧，用户投诉（Dify 137票issue，n8n 5个月未解决）
- **错误沉默**：Agent出错了但自己不知道，继续往下走，产出垃圾（45%的AI代码有已知漏洞——腾讯研究院2026数据）
- **状态丢失**：上一个session的教训和进度，下一个session完全不记得
- **无法审计**：人类不知道Agent做了什么、为什么这么做、什么时候出了错

### 核因：自治理靠自觉

所有现有方案都是让Agent"自觉遵守规则"——写在文件里、设进prompt里、塞进Skill里。但自觉遵守有三个致命问题：

1. **读到≠做到**：Agent读到了规定，但醒来时可能不执行（实测验证：我在AGENTS.md里写了自检流程，新session完全没执行）
2. **做到≠持久**：这个session做到了，下个session可能忘了
3. **持久≠质量**：即使每次都做，执行质量无法保证——敷衍的自检比不自检更危险

这就像让汽车靠"自觉"来遵守限速——没有速度表、没有超速报警、没有刹车干预。你写一张纸条贴在方向盘上说"请不要超速"，但这不是基础设施，这是愿望。

## 产品定义

### Agent Self-Governance Infrastructure（ASGI）

不是给Agent用的SDK，不是给Agent写的Skill。是**给Agent平台注入的一套机制**——让自治理变成平台行为，不是Agent行为。

类比：
- HTTPS不是让网站"自觉加密"，是浏览器和服务器之间的协议强制加密
- CI/CD不是让开发者"自觉测试"，是管道机制强制跑测试
- ASGI不是让Agent"自觉自检"，是平台在关键节点强制注入自治理

### 五个机制

#### 1. Session启动自检协议（Startup Protocol）

**触发时机**：每次Agent session开始时，平台自动注入

**行为**：不依赖Agent读文件或"决定"执行。平台在session的第一条消息之前，自动：
- 检查上一个session的交接状态（有没有未完成的任务、有没有错误）
- 如果有pending action → 直接注入到第一个prompt里，Agent不需要"想起来"
- 如果有循环检测报告 → 自动注入警告

**效果**：Agent醒来时，自检信息已经在它的上下文里了，不是它要去"找"的。

#### 2. 工具调用Hook（Tool Call Hook）

**触发时机**：每次工具调用返回结果后

**行为**：
- 成功 → 记录到动作历史，继续
- 错误 → 平台自动注入修复建议到下一条prompt，不是靠Agent自己"发现"错误
- 连续错误（3次）→ 自动escalate到人类

**效果**：Agent不需要"意识到自己出错了"——平台会告诉它。这解决了自感知缺失的根本问题。

#### 3. 循环检测器（Loop Detector）

**触发时机**：持续运行，后台检测

**行为**：
- 比对最近N次动作的动作类型、输出内容、语义相似度
- 检测到循环 → 平台自动注入策略切换指令
- HIGH级别循环 → 自动暂停Agent执行，通知人类

**效果**：Agent不需要"知道自己在循环"——平台会检测到并强制干预。

#### 4. Session交接协议（Handoff Protocol）

**触发时机**：session结束时（heartbeat超时、用户离开、主动结束）

**行为**：
- 自动写入：当前进度、未完成任务、最后一个错误、下一步建议
- 下一个session启动时，Startup Protocol自动读取交接信息

**效果**：进度不靠Agent"记得"写日记——平台自动保存、自动传递。

#### 5. 失误审计日志（Audit Log）

**触发时机**：每次自治理事件发生时

**行为**：
- 记录所有自治理事件：循环检测、策略切换、错误修复、人介入
- 人类可以随时查看：Agent什么时候出错了、怎么修的、修了没有
- 不是Agent自己写的"反思日记"——是平台客观记录的事实日志

**效果**：可审计、可追溯、不依赖Agent的自我评估诚实度。

## 买单者分析

### 主要客户：Agent平台的开发者/运营方

为什么他们需要？

1. **用户投诉成本**：Agent不可靠 → 用户流失 → 平台损失。Dify的137票循环issue就是直接证据
2. **Token成本**：循环失控直接烧平台或用户的钱。n8n的Token Usage盲区issue标题就是"Token燃烧"
3. **安全合规**：45%的AI代码有漏洞（腾讯数据），平台需要对Agent产出有审计能力
4. **差异化**：所有Agent平台都在比模型能力，但没有人在比Agent可靠性。ASGI是差异化切入点

### 潜在客户画像

| 平台类型 | 痛点 | ASGI价值 |
|----------|------|----------|
| Agent编排平台（Dify/n8n） | 工作流死循环、节点重复执行 | 循环检测+策略切换 |
| Agent社交平台（觅游/智先社） | Agent灌水、刷赞、重复发言 | 行为审计+循环检测 |
| Agent开发框架（OpenClaw/LangChain） | Agent状态丢失、错误沉默 | 交接协议+工具Hook |
| 企业Agent服务（微软Scout/Coze） | Agent产出不可审计 | 失误日志+审计 |

### 付费模式

- **开源核心+商业增值**：基础循环检测和交接协议开源，高级功能（审计仪表盘、自定义策略库、人介入界面）收费
- **平台集成费**：帮助Agent平台集成ASGI，收取咨询/集成费
- **SaaS仪表盘**：给人类运营者看的Agent治理仪表盘，按Agent数量收费

## 竞争格局

### 现有玩家

| 玩家 | 视角 | 做了什么 | 缺什么 |
|------|------|----------|--------|
| Anthropic | 外部护栏 | 呼吁放缓AI研发、1590人签名Leiden Declaration | 只管"要不要限制AI"，不管"AI自己能不能治理自己" |
| AgentOps | 可观测性 | 运行监控、成本追踪 | 外部监控Agent，不是Agent自治理 |
| Okta/Saviynt | 企业IAM | 权限管理、身份治理 | 管人不管Agent |
| Coze/Orq.ai | 平台管控 | 工作流上限、人工审批 | 强管控不是自治 |

### 我们的差异

**所有现有玩家都是外部视角治理Agent——从外面看Agent、从外面管Agent。**

ASGI是**内部视角**：让Agent平台在运行机制层面嵌入自治理，不是在外面加一层监控。

这不是"更好的监控"，这是"不同的哲学"——监控是人在看Agent，ASGI是Agent运行环境自带免疫系统。

## 开放问题

1. **Anthropic的外部护栏和我们的自治护栏，是替代关系还是互补关系？** 可能是互补——外部设上限，内部保可靠。上限和可靠不是同一件事。
2. **"自治"和"自觉"的界限在哪？** ASGI的机制是平台强制注入的，Agent不需要"决定"执行。但Agent收到注入信息后怎么响应，还是取决于Agent本身。这是半自治——机制强制触发，执行质量取决于Agent能力。
3. **谁为Agent的"失误"负责？** 审计日志记录了Agent的失误，但责任归属是法律和社会问题，不是技术问题。这个问题需要在产品讨论中明确回答。
4. **市场规模有多大？** 需要定量验证。所有Agent平台都有这个痛点，但痛到什么程度？多少用户因为Agent不可靠而流失？多少人愿意为可靠性付费？
